# 组合触发条件设计方案 (Features: Combo Trigger)

## 1. 需求背景
当前系统的违规惩罚规则是“单线作战”，即一条规则只能绑定一种特定的违规类型（如迟到、超时、爽约等）。这导致无法应对用户在违规边缘反复试探的行为（例如：迟到2次+超时2次+临期取消2次，完美避开单一规则的惩罚阈值）。
为了堵住这个漏洞，需要引入“组合触发条件”，允许管理员将多种违规类型组合在一起，设定一个总的触发阈值（例如：任意违规累计满 X 次即触发惩罚）。

## 2. 核心设计原则
*   **采用方案 A（多选组合）**：不引入复杂的积分制，保持系统简洁。管理员可以在创建规则时，勾选多个违规类型组成一个“组合”。
*   **不引入核销机制（严厉模式）**：根据需求，受过惩罚的违规记录在统计窗口期（如过去30天）内依然有效。这意味着如果用户解封后仍在窗口期内，只要再犯一次，就会因为总数依然大于阈值而再次触发惩罚。这体现了对屡教不改者的严厉打击。
*   **兼容旧数据**：新的数据结构必须向前兼容，保证系统中已有的单一类型规则不受影响。

## 3. 数据结构调整

### 3.1 数据库表结构
`penalty_rules` 表的结构无需改变，主要修改其 `trigger_config` 字段中存储的 JSON 结构。

### 3.2 JSON 结构扩展
在 `trigger_config` JSON 中，引入新的字段 `violation_types`（数组），以替代或兼容原有的 `violation_type`（字符串）。

**旧结构示例**：
```json
{
  "violation_type": "late",
  "metric": "count",
  "threshold": 3,
  "window_type": "rolling_days",
  "period_days": 30
}
```

**新结构示例（组合触发）**：
```json
{
  "violation_types": ["late", "overdue", "late_cancel"], // 数组，包含所有选中的类型
  "metric": "count",
  "threshold": 5, // 这几种违规类型的总和达到 5 次
  "window_type": "rolling_days",
  "period_days": 30,
  "count_strategy": "by_reservation" // 新增：统计策略
}
```

### 3.3 统计策略 (count_strategy)
为了防止“一次预约同时迟到和超时”导致瞬间触发多次违规（显得过于严苛且不合理），引入 `count_strategy` 字段：
*   `by_record`（默认/严格模式）：按违规记录的行数统计。迟到1次+超时1次 = 2次违规。
*   `by_reservation`（宽容模式/按事件去重）：按 `reservation_id` 去重统计。同一次预约中无论触发了多少种选中的违规类型，都只算作 1 次违规事件。

## 4. 后端规则引擎适配 (`server.ts`)

核心函数 `evaluatePenaltiesOnViolation` 需要进行以下调整：

1.  **解析兼容**：读取规则时，优先读取 `trigger.violation_types`（数组）。如果不存在，则回退读取旧的 `trigger.violation_type` 并将其转化为单元素数组 `[trigger.violation_type]`，以保证兼容性。
2.  **SQL 查询构建**：
    *   将原来的 `violation_type = ?` 替换为 `violation_type IN (?, ?, ...)`。
    *   动态生成 `IN` 语句的占位符，并将数组展开推入 `queryParams`。
3.  **去重逻辑实现**：
    *   如果 `trigger.count_strategy === 'by_reservation'` 且 `metric === 'count'`，SQL 查询应使用 `COUNT(DISTINCT reservation_id)` 或在代码层面按 `reservation_id` 进行去重统计。
    *   如果 `metric === 'duration'`，则直接对所有符合条件的记录的 `duration_minutes` 求和（时长通常不需要去重）。

## 5. 前端 UI 交互设计 (`ViolationsAndPenaltiesTab.tsx`)

在“新建/编辑规则”的弹窗表单中进行调整：

1.  **违规类型选择器**：
    *   将原有的 `<select>` 单选框替换为一组 Checkbox（复选框），或者一个支持多选的下拉组件。
    *   选项包括：迟到、超时、爽约、临期取消。
    *   管理员可以勾选一个或多个。至少需要勾选一个。
2.  **统计策略选择器（新增）**：
    *   当勾选了**多个**违规类型时，显示一个额外的设置项：“同一次预约触发多种违规时如何计算？”
    *   选项 A：“分别计算（例如：迟到且超时算2次违规）” -> 对应 `by_record`
    *   选项 B：“合并计算（同一次预约最多算1次违规）” -> 对应 `by_reservation`
3.  **文案动态更新**：
    *   当勾选多个类型时，阈值设定的文案应动态变为：“当上述勾选的违规行为累计达到 [ X ] 次时触发”。

## 6. 边缘情况处理

*   **无核销机制的连环触发**：由于明确要求不引入核销机制，如果用户触发了“封禁 7 天”的惩罚，解封后，只要历史违规记录还在 30 天的窗口期内，系统在下一次扫描或用户下一次违规时，会再次计算总数。如果总数依然大于阈值，会**立刻再次触发封禁**。这是符合“严厉模式”预期的行为，无需在代码中做特殊拦截（除了现有的“防止在封禁期内重复插入相同的 active penalty”逻辑）。
*   **规则重叠**：如果管理员设置了“迟到满3次封禁”和“组合违规满5次封禁”，用户可能同时触发两个规则。系统现有的逻辑是遍历所有规则，满足条件就插入 `user_penalties`。这会导致用户同时背负两个惩罚记录，这也是允许的（惩罚时间可能会重叠或顺延，取决于前端展示和预约拦截逻辑）。
