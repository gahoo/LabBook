# 方案 B：后端直返细节与成功后展示限制

## 背景

当前违规拦截流程：前端发起预约 `POST /api/reservations` -> 后端校验到被封禁 (BAN) 则返回 403 (带简单的错误字符串) -> 前端通过检测 403 错误文本中的“因触发”关键词，发起二次请求 `/api/violations/my` 获取触发规则 -> 渲染拦截弹窗。

这样的流程存在以下不合理之处：
1. **强耦合的逻辑判断**：前端依赖文本匹配（`error.includes('因触发')`）来决定下一步操作，非常脆弱。
2. **多余的网络请求**：拦截时本已得出受限结果，却不直接传递，要客户端二次抓取。
3. **附带惩罚显示不明**：如“费用加倍”等非拦截(非 BAN) 惩戒只在扣费逻辑中起效，前端预约过程中没有直观传达给用户。

## 方案 B 核心逻辑

通过将状态提示从表单验证阶段移除，转移至提交阶段响应及最终界面，降低 UI 的复杂性。

**1. 拦截时 (BAN状态) 后端 403 直返详情**
当用户被判定为封禁不允许预约时，在提交 `/api/reservations` 请求后，后端直接附带详细受限上下文：
```json
{
  "error": "因处罚规则被限制...",
  "penaltyDetails": {
    "penaltyMethod": "BAN",
    "reason": "...",
    "violation_ids": [...],
    "triggered_rules_details": [...]
  }
}
```
前端捕获到 403 并有 `penaltyDetails` 时，直接渲染弹窗，无需再次请求违规接口。

**2. 预约成功时不拦截，界面延后提示额外惩罚 (需审批/加倍)**
为避免在填表时需要维护过多的实时验证逻辑：
- 提交预约成功时返回 200，后端在响应体中将当次的 `penaltyCheck`（或实际发生的翻倍、审批状态）包含在内或写入预约结构中（例如包含在 `reservation.penalty_notice` 等字段中）。
- 跳转至“预约凭证 / 预约码展示”组件界面时，在这个显眼的位置单独展示警告横幅：
  - **需审批提示**：“因触发【...】规则，本次预约为**待审批**状态，需管理员通过后方可生效。”
  - **费用加倍提示**：“因触发【...】规则，本次预约的费用将按 **{X}倍** 结算。”

## 优缺点分析与对比 (对比前端实时拦截提示)

### 优点
1. **极简的前端状态管理**：预约表单仅负责处理正常流程，不需要在用户选定时间、人员和仪器的同时，实时轮询或请求服务端检查“当前条件组合下我会不会被惩罚”。
2. **减少竞态和网络开销**：拦截动作在最终提交时一次性校验，错误弹窗依赖单次 HTTP 的完整数据，避免二次请求和加载等待。
3. **避免 UI 侵入**：成功页本身就是一个结果展示页，在上面增加惩罚提示比在复杂的表单上插槽更容易。

### 缺点
1. **反馈时机后置（体验硬伤）**：这是针对“加倍扣费”而言。用户在表单填写时毫不知情，点完「确认提交」后才突然看到提示“本次费用加倍”。如果用户觉得不划算而去取消，可能又恰好触发了“临期取消”规则。
2. **API 接口的数据结构需扩展**：目前的创建预约 API 返回可能是记录对象的数组，我们需要向里面附带额外的属性或重新包装外层 JSON 结构，并确保前端能平滑兼容。
